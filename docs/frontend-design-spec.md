# 前端设计与实现统一规范

## 1. 文档目的

本文档定义 Nexus 前端在 `launcher` 与 `/app` 两大界面中的统一设计原则、页面叙事方式、表面层级规则、动效配额、内容输入要求，以及 Codex 参与前端实现时必须遵守的工作流。

这份规范不是“写给模型的提示词合集”，而是前端实现的硬约束。后续所有 UI 重构、页面新增、视觉改版和组件抽象，都必须先满足本文档，再讨论局部样式。

若具体页面实现与本文档冲突，应优先收敛到本文档定义的结构纪律与信息层级。

## 2. 适用范围

本文档覆盖：

- `launcher` 首屏与其右侧应用面板
- `/app` 工作台全部页面
- 共享 Header / Dialog / Card / Input / Avatar / Toolbar 语法
- 组件抽象与 Codex 生成前端时的执行规则

本文档不覆盖：

- 后端 API 设计
- 数据模型与状态流协议
- 营销官网或第三方站点

## 3. 总体设计目标

Nexus 前端必须长期满足以下目标：

1. `launcher` 是情绪化入口，`/app` 是克制的工作台，两者不能混用同一视觉语言
2. 页面必须先有视觉立场与信息结构，再进入组件实现
3. `/app` 必须像桌面工具，而不是网页卡片集合
4. 真正的 glass / refraction 只能服务于少数关键表面，不能扩散到小控件
5. 所有页面必须优先保证内容可读性，而不是效果存在感
6. Codex 生成前端时，必须先被约束，再生成代码

## 4. 几类概念先分开

### 4.1 视觉主张

- 表示某个页面首屏要传达的整体气质
- 例如：冷静、精确、协调、轻盈、系统感
- 它不是颜色说明，也不是组件清单

### 4.2 内容结构

- 表示页面如何按“首屏承诺 → 支撑信息 → 细节说明 → 收口动作”组织
- 它不是简单文档目录
- 每个区块只能承担一个主要任务

### 4.3 交互主张

- 表示页面只保留哪些必要动效
- 通常全页最多 2 到 3 类
- 它不是“能动就动”

### 4.4 表面层级

- 表示用户最终看到的背景、主面、次级卡片和浮层之间的层级关系
- 它只关心“用户能感知到的面”
- 它不等于 DOM 层数，但 DOM 结构必须服务于它

### 4.5 组件语法

- 表示按钮、标签、卡片、头像、输入框、弹窗这些共享元素的固定表达方式
- 一旦定义，页面不得随意改写

## 5. 页面实现前的三项前置产物

任何新页面、重构页或大改版，在动手写代码前必须先明确三项内容：

### 5.1 视觉主张

只允许一句话，说明页面气质。

示例：

- `launcher`：一个安静但具有召唤感的 AI 协作入口
- `/app`：一个信息密集但克制清晰的多任务工作台

### 5.2 内容结构

必须按页面叙事写出最少这几层：

1. 首屏建立身份或任务
2. 支撑信息说明当前上下文
3. 细节区承载主要操作
4. 收口区给出下一步动作

如果一个区块承担两个以上任务，必须拆开。

### 5.3 交互主张

每页最多保留 2 到 3 个动效类别：

- 一个入场节奏
- 一个滚动或吸附行为
- 一个 hover / reveal / layout transition

超出三类的动效默认视为噪音，直接删除。

## 6. `launcher` 与 `/app` 的角色分离

### 6.1 `launcher`

`launcher` 的职责是：

- 建立品牌与主智能体身份
- 承接首次输入
- 快速把用户带入协作

因此它允许：

- Hero 构图
- 少量情绪化材质
- 受控的入口动画
- 一个主要视觉焦点

但它不允许：

- 多个同等级视觉焦点并存
- 底部和侧边同时争抢注意力
- Hero 区里堆一排卡片
- 过多的局部 glass 小控件

### 6.2 `/app`

`/app` 的职责是：

- 快速理解当前工作上下文
- 高密度展示真实信息
- 支持连续操作

因此它必须：

- 优先使用清楚的标题、标签、数字和状态
- 使用克制的表面层级
- 保持稳定的工具感

它不允许：

- 为了“好看”增加多余的大面板
- 页面像营销页一样追求构图感
- 小按钮、小 badge、小 tab 全部使用强视觉材质

## 7. 页面层级预算

### 7.1 `/app` 层级预算

`/app` 的最终可感知层级最多只允许以下四层：

1. 全局背景
2. Sidebar rail
3. 主内容 plane
4. 浮层 / Dialog / Popover

禁止出现：

- 全局背景之上再叠一个“页面背景板”
- 主内容区内部再套一层“整页 panel”
- section 再套 section
- inset 再套 inset

### 7.2 `launcher` 层级预算

`launcher` 可比 `/app` 多一层 Hero 构图面，但也必须克制：

1. 全局背景
2. Hero 主要构图面
3. 右侧或辅助信息面
4. 浮层

禁止 Hero 内再继续堆多层局部面板。

## 8. 表面语法统一规则

### 8.1 `/app` 只允许三种主语法

- `rail`
- `plane`
- `card`

解释如下：

- `rail`：侧边导航承载面
- `plane`：页面主内容承载面
- `card`：真正有独立信息单元时才使用

禁止长期并存以下近义表面：

- panel
- shell
- subtle panel
- inset shell
- glass card
- modal card

如果语义不同但视觉接近，必须继续合并。

### 8.2 Card 使用规则

只有在“这个卡片本身就是信息单元或操作单元”时才允许使用 `card`。

以下场景不应使用卡片：

- Hero 区背景承载
- 页面工具条
- 只是为了把内容框出来
- 输入框外层
- badge 的承载壳

### 8.3 状态与动作必须分离

状态只能用：

- 文本
- 轻 badge
- 状态点

动作只能用：

- 文本按钮
- 图标按钮
- 主动作按钮

禁止把“未配置”“已导入”“已连接”“可安装”这类状态伪装成按钮。

## 9. Glass 与材质规则

### 9.1 真折射使用范围

真折射只允许出现在：

- `launcher` 的 Hero 主输入面
- 一个系统级高优先浮层

除此之外，全部退回静态材质。

### 9.2 小控件禁止使用真 glass

以下元素禁止使用真折射或强玻璃：

- tabs
- badge
- chip
- avatar
- 小型按钮
- 列表项
- 工具栏动作

这些元素只允许使用静态、轻量、可读的实体材质。

### 9.3 Dialog 必须是实面

所有 Dialog 必须遵守：

- 遮罩负责隔离背景，不负责制造效果
- 主体必须是近实底
- Header / Body / Footer 要能独立读清
- 内部卡片数量要尽可能少

Dialog 的可读性优先级高于所有视觉实验。

### 9.4 Avatar 必须是实体

头像只承担身份识别，不承担氛围效果。

头像必须：

- 有明确边界
- 有稳定前景对比
- 与背景分离

禁止把头像做成半透明玻璃球。

## 10. Hero 规则

### 10.1 首屏必须先读成一个整体

首屏的第一感受必须是“一个构图”，不是“一堆组件”。

判断标准：

- 隐去细节后仍能读出主视觉焦点
- 第一屏只存在一个主要承诺
- 视线不会在三个以上焦点间跳跃

### 10.2 Hero 区禁止卡片堆叠

Hero 里不允许出现：

- 功能卡片网格
- 多个同级玻璃卡
- 用卡片模拟文案分区

Hero 应该使用：

- 单一主面
- 受控辅助信息
- 明确的品牌或主动作

### 10.3 品牌优先

在品牌页或入口页：

- 产品名必须具备 Hero 级别
- 品牌不可退化成角落角标
- 如果页面去掉图片后仍不清楚自己是谁，说明品牌表达不足

## 11. `/app` 工作台规则

### 11.1 默认为工具界面，而非展示页

`/app` 页面应接近 Linear 一类产品的节奏：

- 平静的表面层级
- 强排版
- 少颜色
- 高信息密度
- 易扫描

### 11.2 工具栏优先于装饰

`/app` 中的 Header / Toolbar 必须优先承担：

- 当前页面身份
- 当前上下文
- 当前可执行操作

而不是：

- 大面积视觉装饰
- 一排胶囊按钮
- 为了“丰富”而加的次级面板

### 11.3 输入区必须是一体化工具带

聊天输入区应默认表现为一个统一工作带，而不是：

- 输入框一层
- 附件一层
- 发送按钮一层
- 提示条一层

它必须被理解为一个连续的工作面。

## 12. 文案与内容输入规则

### 12.1 必须使用真实内容

前端生成时，必须尽量提供：

- 真实页面标题
- 真实说明文案
- 真实业务标签
- 真实状态名称

占位文案会直接导致占位式布局。

### 12.2 页面必须有明确任务语义

页面不是文档，而是任务界面。

每个区块都必须回答：

- 用户现在在看什么
- 为什么要看
- 下一步要做什么

如果回答不出来，这个区块应当删除或重写。

## 13. 动效规则

### 13.1 动效配额

每页最多 3 类动效：

1. 入场
2. 滚动或吸附
3. hover / reveal / layout transition

### 13.2 动效优先级

动效必须优先表达：

- 层级
- 焦点
- 状态变化

而不是“页面显得更活”。

### 13.3 技术规则

优先使用：

- `opacity`
- `translate`
- `scale`

谨慎使用：

- `filter`
- `blur`
- 大范围 `backdrop-filter`
- 多层持续动画

如果一个动效只提供装饰价值，应删除。

## 14. Codex 参与前端实现的执行规则

### 14.1 先约束，后生成

Codex 参与页面实现时，输入必须至少包含：

1. 页面类型：`launcher` / `app` / `dialog` / `tool page`
2. 视觉主张
3. 内容结构
4. 交互主张
5. 真实内容
6. 至少一个视觉参考

缺少这些约束时，模型默认会回到训练集中最常见的安全模板。

### 14.2 推理强度

前端生成默认使用 `low` 或 `medium` 推理强度。

原因：

- 更容易保持节奏和整体感
- 不会因为过度分析把页面写成工程拼贴
- 更利于快速迭代与回看

### 14.3 生成顺序

Codex 生成前端时必须遵守以下顺序：

1. 先确定页面叙事
2. 再确定布局骨架
3. 再确定共享组件语法
4. 最后才写具体代码

禁止一上来直接开始拼 JSX。

## 15. 组件设计纪律

### 15.1 Header

Header 只负责：

- 页面身份
- 当前状态
- 少量动作

Header 不应成为第二个 Hero。

### 15.2 Dialog

Dialog 只负责：

- 让用户完成一件事
- 提供必要上下文
- 给出明确确认与取消

如果一个 Dialog 需要分成多个页面式区域，说明它不该是 Dialog。

### 15.3 Catalog Card

所有目录卡片必须遵守统一语法：

- 头部身份
- 主体说明
- 底部状态 / 动作

页面不能自己再发明一套卡片壳。

### 15.4 Toolbar Action

Toolbar 动作必须是轻量语法，不得做成主视觉元素。

### 15.5 Badge

Badge 是状态，不是按钮，不是装饰。

## 16. 视觉验收标准

任何页面上线前必须通过以下检查：

### 16.1 首屏检查

- 第一屏是否先读成一个整体
- 是否只有一个主要焦点
- 是否能在 3 秒内理解页面用途

### 16.2 层级检查

- 是否存在多余的大面板
- 是否存在面板套面板
- 是否有组件为了“被看见”而额外加一层壳

### 16.3 可读性检查

- 只看标题、标签、数字和按钮，能否理解页面
- 弹窗正文是否清楚
- 头像和状态是否与背景分离

### 16.4 动效检查

- 是否只有 2 到 3 类动效
- 动效是否承担了层级或状态表达
- 是否存在只为装饰而动的元素

## 17. 迁移与收敛规则

### 17.1 新页面不得带入旧语法

任何新页面不得再引入：

- 新的 glass recipe
- 新的 panel 命名
- 私有卡片样式系统
- 私有按钮视觉语言

### 17.2 老页面优先做减法

已有页面优化时，优先顺序必须是：

1. 砍层级
2. 合并语法
3. 收实材质
4. 最后才做视觉润色

### 17.3 共享层优先级最高

如果局部页面和共享组件系统冲突，优先修改共享组件，而不是在页面里继续打补丁。

## 18. 结论

Nexus 前端的目标不是“做出更炫的页面”，而是建立一套可持续的设计纪律：

- `launcher` 有明确入口气质
- `/app` 像真正的工作台
- 表面层级可控
- 动效有配额
- 内容优先
- 模型生成必须被约束

后续所有前端改动，都应以本文档作为设计与实现的统一基线。
